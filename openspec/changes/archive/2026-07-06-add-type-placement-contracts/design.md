## Context

Every existing contract family checks *references* (source layer/type A references forbidden namespace/group/assembly B) and reuses the single `ArchitectureViolation` record shape (`SourceType`, `ForbiddenNamespace`, `ForbiddenReferences`, plus optional per-family fields), dispatched through `ArchitectureContractExecutor` via the catalog/handler-registry seam with zero executor changes per family (most recent precedent: `assembly_allow_only`/`external_allow_only`, #58/#60). Type metadata across all contracts today is reflection-based (`System.Type` via `ArchitectureTypeIndex`/`ArchitectureTypeScanner`), not Roslyn-symbol-based — the project-aware Roslyn analysis added in #61 is narrowly scoped to method-body call resolution only and exposes no general type-metadata query API.

Issue #85 asks for something structurally new: a contract that doesn't check *what a type references*, but *where a type itself lives* and *what it's named*. There is no reference/target-namespace concept here at all.

## Goals / Non-Goals

**Goals:**
- One contract family, `type_placement`, that selects types via constrained matchers (name suffix/prefix, namespace, layer, base type, interface, attribute) combined with AND semantics, and checks two independent expectations against the selection: placement (must reside in a declared layer/namespace/project/assembly) and naming (required/forbidden suffix/prefix).
- Reuse the existing reflection-based type index; add matcher logic (base type chain walk, interface list, attribute data) as a focused new helper, not a new metadata subsystem.
- Reuse the existing `ArchitectureViolation`/diagnostic-mapper pattern (new optional fields + new diagnostic record + new `ArchitectureDiagnosticKind`) rather than inventing a parallel violation pipeline — this keeps `ArchitectureContractExecutor` and `ArchitectureHandlerResult` completely unchanged.
- Deterministic output: matcher AND-combination, alphabetically/ordinally stable iteration, one violation per (type, contract).

**Non-Goals:**
- Regex or any expression language for matchers (issue explicit non-goal). Matchers are exact/prefix/suffix/full-name string comparisons only.
- True Type→`.csproj` mapping. `must_reside_in_projects` is sugar over assembly-name matching via `ArchitectureDiscoveredProject.AssemblyName` — documented explicitly, not hidden.
- Method-body IL scanning. Placement/naming are structural properties of the type declaration itself, not usage-site concerns.
- Runtime DI resolution, semantic data-flow analysis, auto-move/auto-rename of files (issue explicit non-goals).
- Any change to existing contract families' semantics — purely additive.

## Decisions

**Single family for placement + naming, not two.**
The issue's own example YAML (`strict_type_placement` with both a selector and a placement expectation) frames this as one family. Placement and naming share the identical selector concept (`types_matching`) and the issue's acceptance criteria test both under the same contract shape. Splitting into two families would duplicate the entire matcher schema for no behavioral benefit and contradicts the issue's own proposed shape.
- Alternative considered: separate `type_placement` and `type_naming` families. Rejected — doubles schema surface, doubles selector-matching code, and the issue never asks for independent audit/strict toggling between placement and naming for the same role (a `Controller` rule wants both checked together).

**Matchers combine with AND; each matcher field checks one property.**
`name_suffix`/`name_prefix` match `Type.Name` (simple name, ordinal). `namespace` reuses `ArchitectureTypeIndex.FindTypesInNamespace`/`ArchitectureLayerResolver.MatchesPrefix` (segment-aware prefix, consistent with every other namespace matcher in the codebase). `layer` resolves a configured layer name via `ArchitectureLayerResolver.ResolveLayer` then `MatchesNamespace`. `base_type` walks `Type.BaseType` chain comparing `FullName` (ordinal, exact — no wildcard) until `null` or a match. `implements_interface` checks `Type.GetInterfaces().Any(i => i.FullName == configured)`. `has_attribute` checks `Type.GetCustomAttributesData().Any(a => a.AttributeType.FullName == configured)`, wrapped the same defensively as `ArchitectureTypeScanner.GetLoadableTypes` (skip types that throw on reflection). All specified fields must match (AND); an unset/empty field is skipped, not treated as "match everything."
- Alternative considered: OR-across-matchers ("any" semantics). Rejected — a policy author writing `name_suffix: Controller` + `layer: api` almost always means "types named `*Controller` that are already in the api layer" (a scoping combination), not "types either named `*Controller` OR in `api`," which would make the matcher far less useful for narrowing scope. AND matches how filters compose in every other contract's field set (e.g., `source` + `forbidden` are both required, not either/or).

**Placement: union across all four `must_reside_in_*` lists, not per-field independent checks.**
A matched type's actual location (layer match OR namespace prefix match OR assembly-name match OR project-resolved-assembly-name match) must satisfy at least one entry across the combined declared set. This mirrors "must live in one of these places," which is the natural reading of the issue's `must_reside_in_layers: [api]` example generalized to multiple residency axes.
- Alternative considered: requiring the type to satisfy *every* non-empty `must_reside_in_*` list independently (AND across list-types). Rejected — a rule declaring both `must_reside_in_layers: [api]` and `must_reside_in_assemblies: [MyApp.Api]` almost always means "either expressing constraint is sufficient/redundant restatement of the same location," not "must simultaneously be in layer api AND assembly MyApp.Api" (which would be true anyway if the layer-to-assembly mapping holds, but is a needless double constraint if authors mix axes). Union keeps the four fields as alternative ways to express the same allowed-location set.

**`must_reside_in_projects` is assembly-name equivalence via project discovery, documented explicitly.**
Confirmed via prior investigation: `ArchitectureDiscoveredProject` maps 1:1 to a single `AssemblyName`; there is no reflection-level Type→project mapping anywhere in this codebase (assembly/project resolution elsewhere, e.g. `ArchitectureAnalysisSession.Coverage.cs`'s `ResolveProjectAssembly`, is already exact-string assembly-name matching against a project's `AssemblyName`). Building true physical `.csproj`-membership tracking for every type would require new Roslyn/MSBuild plumbing far beyond this issue's scope. `must_reside_in_projects` therefore resolves each configured project name to its `AssemblyName` via project discovery, then checks it identically to `must_reside_in_assemblies`. This is called out explicitly in `docs/contracts/type-placement.md`'s scope section, not silently glossed over.
- Alternative considered: omit `must_reside_in_projects` entirely for v1, ship only `must_reside_in_assemblies`. Rejected — the issue's acceptance criteria explicitly say "layers, namespaces, projects, or assemblies," and the equivalence is a real, honest capability (most single-project-per-assembly .NET solutions get exactly what they'd expect), just documented with its actual semantics rather than an idealized one.

**Diagnostics: extend `ArchitectureViolation` with four optional fields; add a dedicated `TypePlacementDiagnostic`.**
`ExpectedTypeLocation`/`ActualTypeLocation` (populated when a placement check fails) and `ExpectedTypeName`/`ActualTypeName` (populated when a naming check fails) — both pairs can be populated on the same violation if a type fails both placement and naming simultaneously. `SourceType` carries the type's full name as usual. The required `ForbiddenNamespace`/`ForbiddenReferences` fields are populated with fallback values derived from the new fields (`ForbiddenNamespace = ExpectedTypeLocation ?? ExpectedTypeName ?? ""`, `ForbiddenReferences = [ActualTypeLocation ?? ActualTypeName ?? SourceType]`) so any generic/legacy formatting path that doesn't yet know about the new kind still renders something sane. `ArchitectureDiagnosticMapper.FromViolation` gets a new branch (checked before the final `DependencyDiagnostic` fallback, alongside the existing `ForbiddenExternalGroup`/`ForbiddenPackageGroup` branches) that dispatches to `TypePlacementDiagnostic` when either new field pair is set. `ArchitectureDiagnosticFormatter` gets human-text and CI-JSON rendering for the new kind (e.g. `- [ContractName] SourceType: expected {location/name}, actual {location/name}`).
- Alternative considered: build a fully separate violation/diagnostic pipeline outside `ArchitectureViolation`/`ArchitectureHandlerResult`. Rejected — `ArchitectureContractExecutor` and every handler signature is hard-coded to `List<ArchitectureViolation>`; a parallel shape would require executor changes, breaking the "zero executor changes per new family" invariant that every prior contract family has preserved.

**Load-time validation: reject a `types_matching` selector with no placement or naming expectation.**
Mirrors the existing convention (e.g. `strict_allow_only` rejecting a group with no matchers) — `ArchitecturePolicyDocumentLoader` gains a check that a `type_placement` contract declares at least one non-empty `must_reside_in_*` list or one non-empty naming field; otherwise it's a configuration error, not a silently-no-op rule.

## Risks / Trade-offs

- **[Risk] `must_reside_in_projects` doesn't catch multi-assembly or assembly-splitting-within-one-project setups.** → Documented explicitly as assembly-name equivalence in `docs/contracts/type-placement.md`; policy authors needing precise physical-file placement guidance should combine this with code review, not rely solely on this contract. Matches the issue's own non-goal ("no semantic data-flow analysis").
- **[Risk] `has_attribute`/`base_type`/`implements_interface` require the attribute/base/interface's declaring assembly to be loadable via reflection.** → Consistent with every other reflection-based matcher already in this codebase (`ArchitectureTypeScanner.GetLoadableTypes` already tolerates `ReflectionTypeLoadException`); the new matcher helper applies the same defensive handling, so this is a known, already-precedented characteristic of the reflection-based approach, not a new failure mode.
- **[Risk] Combining placement+naming into one violation could be seen as "less granular" than two.** → Mitigated by carrying both field pairs independently on the same `ArchitectureViolation`/diagnostic so a formatter/consumer can still report on them individually; the issue's acceptance criteria only require the diagnostic to "identify the type, matched role rule, expected location/name, and actual location/name" — singular combined wording, not "two diagnostics."

## Migration Plan

Purely additive: new YAML contract-group keys, new schema defs, new docs page, four new optional `ArchitectureViolation` fields (existing consumers unaffected since they're additive/optional), one new `ArchitectureDiagnosticKind` enum value (additive). No existing contract family, schema field, or CLI behavior changes. No migration or rollback beyond normal code review/revert.
