## Context

`ArchitectureDependencyContract` (`strict`/`audit`, `dependency-contracts` capability) and `ArchitectureAllowOnlyContract` (`strict_allow_only`/`audit_allow_only`, `allow-only-contracts` capability) already give directional, namespace/layer-scoped forbidden and allow-list checks via `ArchitectureAnalysisSession.CheckContract`/`CheckAllowOnlyContract` (`ArchitectureAnalysisSession.Checking.cs`). `ArchitectureAssemblyIndependenceContract` (`strict_assembly_independence`/`audit_assembly_independence`, #51) added the first assembly-axis check, but only mutual independence ("none of these may reference any other") via `CheckAssemblyIndependenceContract` (`ArchitectureAnalysisSession.AssemblyIndependence.cs`) — it has no concept of a single directional source, a forbidden list, or an allow-list.

Issue #58 asks for directional assembly/project contracts: `source` must not reference `forbidden` (or, the allow-only inverse: `source` may only reference `allowed`). This must be added through the same contract-catalog + handler-registry pipeline #51 used (`contract-handler-execution` capability): a family becomes executable by (1) adding a model + YAML group, (2) registering it in `ArchitectureContractCatalog.Build`, (3) implementing an `IArchitectureContractHandler`, and (4) one DI registration per handler — `ArchitectureContractExecutor` dispatches every family generically over `FamiliesInOrder` with no changes required.

`Context.TargetAssemblies` (on `ArchitectureAnalysisSession.Context`) already holds every assembly resolved from `analysis.target_assemblies`, matched by `assembly.GetName().Name`. `Assembly.GetReferencedAssemblies()` gives direct assembly-level references for free from these already-loaded assemblies, exactly as #51 used it — no new `.csproj`/`.deps.json` parsing is needed for this change.

## Goals / Non-Goals

**Goals:**
- Let a policy declare `strict_assembly_dependency`/`audit_assembly_dependency` contracts naming one source assembly and a list of forbidden assemblies it must not directly reference.
- Let a policy declare `strict_assembly_allow_only`/`audit_assembly_allow_only` contracts naming one source assembly and a list of assemblies it may directly reference; any other direct reference to a *declared* (`analysis.target_assemblies`) assembly is a violation.
- Produce deterministic `ArchitectureViolation`s identifying the source assembly, the forbidden/disallowed target assembly, and the contract id/name, distinguishable from namespace/layer, `assembly_independence`, and `.asmdef` violations by contract family/group.
- Implement entirely through the existing contract catalog/handler seam, with zero changes to `ArchitectureContractExecutor`, the CLI, or the public validator beyond wiring supplied by that seam.
- Fail loudly at policy-load time if any assembly name referenced by these contracts (`source`, `forbidden`, `allowed`) isn't resolvable via `analysis.target_assemblies`, mirroring #51's `ValidateAssemblyIndependenceContracts`.
- Leave `strict`/`audit`, `strict_allow_only`/`audit_allow_only`, `strict_assembly_independence`/`audit_assembly_independence`, and `.asmdef` checks byte-for-byte behaviorally unchanged.

**Non-Goals:**
- Transitive assembly reference path resolution — both new families check only direct references (`Assembly.GetReferencedAssemblies()`), matching #51's decision for `assembly_independence`. A `dependency_depth`-style transitive option (mirroring `ArchitectureDependencyContract`) is deferred; walking transitive assembly references would require resolving assemblies beyond `Context.TargetAssemblies` (referenced assemblies aren't guaranteed to already be loaded/declared), which is materially more complex than the namespace-level transitive walk (which stays within already-scanned types) and is not needed to satisfy #58's acceptance criteria.
- Ordered assembly/project layer contracts (an assembly-axis analog of `ArchitectureLayerContract`/`strict_layers`) — explicitly deferred to a follow-up change, matching how `layer` is already a separate family from `dependency`/`allow_only` at the namespace level.
- Assembly/project cycle detection (an assembly-axis analog of `ArchitectureCycleContract`/`strict_cycles`) — explicitly deferred to a follow-up change, matching how `cycle` is already a separate family from `dependency`/`allow_only` at the namespace level.
- Parsing `.csproj`/`ProjectReference` elements or `.deps.json` — resolution is via already-loaded reflection `Assembly` objects, same as every other contract family.
- Any interaction with the namespace-independence-driven dependency-edge coverage mechanism (`ArchitectureAnalysisSession.Coverage.cs`) — that mechanism is a namespace-axis concern; extending coverage to the new assembly-axis families is out of scope here.
- Any changes to Unity `.asmdef` scanning or `assembly_independence`.

## Decisions

**1. Two contract families, not one, mirroring the namespace-level split between `dependency` and `allow_only`.**
`ArchitectureDependencyContract` (forbidden list) and `ArchitectureAllowOnlyContract` (allowed list) already exist as separate families at the namespace level because their semantics differ: forbidden-list violations are explicit and bounded by the contract's own list, while allow-list violations require knowing the full universe of "things that count" so anything not on the list can be flagged. Introducing a single combined model with both `forbidden` and `allowed` fields (only one populated) would require runtime branching on which field is set and would break the existing 1:1 pattern between YAML group and model type that every other family in this codebase follows. Two distinct models/groups/handlers keeps each family's YAML shape unambiguous and matches `dependency`/`allow_only`'s existing precedent exactly.

**2. `assembly_dependency` — single `source` name + `forbidden` list, not `assembly_independence`'s flat `assemblies` list.**
`assembly_independence` uses one flat list because every pair is checked both ways (mutual). Directional forbidden checks need a distinguished source, exactly like `ArchitectureDependencyContract`'s `source`/`forbidden` shape. Model: `ArchitectureAssemblyDependencyContract { Name, Id, Source: string, Forbidden: List<string>, IgnoredViolations, Reason }`.
*Alternative considered*: reuse `assembly_independence`'s `assemblies: string[]` shape with an implicit "first entry is source" convention. Rejected — implicit positional meaning is exactly the kind of "magic" this codebase avoids elsewhere (every other directional family has an explicit `source` field); an explicit field is unambiguous in YAML and in code.

**3. `assembly_allow_only` — single `source` name + `allowed` list; violations scoped to *declared* target assemblies, not every direct reference.**
Mirroring `ArchitectureAllowOnlyContract`'s namespace-level behavior (which excludes references to types outside any declared layer, i.e. "not project types," from allow-only violations — see `allow-only-contracts` capability, "Allow-only ignores non-project types"), the assembly-level allow-only check only flags a direct reference to another assembly if that assembly is itself in `analysis.target_assemblies` (the declared, in-scope assembly universe) and not in `allowed` (or the source itself). Direct references to assemblies outside `analysis.target_assemblies` (framework/BCL/NuGet assemblies never declared to ArchLinterNet) are not violations — they aren't "project assemblies" any more than `System.String`'s namespace is a "project type" at the namespace level.
*Alternative considered*: flag every direct reference not in `allowed`, including framework/NuGet assemblies. Rejected — this would make `assembly_allow_only` unusable in practice (every assembly transitively references BCL assemblies) and has no namespace-level precedent; the namespace-level allow-only contract explicitly scopes violations to "project types" for the same reason.

**4. Direct assembly-reference detection via `Assembly.GetReferencedAssemblies()`, matched by simple name, no transitive walk — same as #51.**
For `assembly_dependency`, for each entry in `contract.Forbidden` (declaration order), the check resolves both `source` and the forbidden name to `Assembly` instances via a `Context.TargetAssemblies` dictionary keyed by simple name (identical resolution to `assembly_independence`) and tests whether the forbidden name appears in `source.GetReferencedAssemblies()`. For `assembly_allow_only`, the check enumerates `source.GetReferencedAssemblies()` once, filters to names present in `Context.TargetAssemblies` and not in `allowed`/self, and reports the remainder (sorted, deduplicated) — mirroring `CheckAllowOnlyContract`'s `OrderBy`/`Distinct` determinism pattern.
*Alternative considered*: building a full transitive reference graph. Rejected for the reasons in Non-Goals — matches the issue's explicit "decide direct vs. transitive" ask, and can be added later as an additive `dependency_depth`-style option without breaking the direct-only shape (same escape hatch #51 designed in).

**5. New loader validation: every assembly name referenced by either family (`source`, `forbidden`, `allowed`) must exist in `analysis.target_assemblies`.**
Identical rationale to #51's `ValidateAssemblyIndependenceContracts`: an unresolvable assembly name is a policy authoring mistake, not a runtime-skippable condition, and validating at load time turns it into one deterministic, actionable error instead of a silently-empty (falsely passing) check. Implemented as two new loader methods (`ValidateAssemblyDependencyContracts`, `ValidateAssemblyAllowOnlyContracts`) alongside the existing `ValidateAssemblyIndependenceContracts`, rather than modifying that method, to keep each family's validation independently readable and to avoid coupling three unrelated model shapes into one generic loop.

**6. Reuse `ArchitectureViolation`/`ArchitectureIgnoredViolation`; distinguish by family/group, not by a new violation subtype.**
Identical rationale to #51 Decision 1: no consumer needs to distinguish violation shape by family at the type level, and every existing family already varies in what `SourceType`/`ForbiddenNamespace` mean. `assembly_dependency` and `assembly_allow_only` violations carry assembly simple names in these fields, exactly like `assembly_independence`, and are distinguished from namespace/layer/`assembly_independence`/`asmdef` violations purely by which family/group produced them (`ArchitectureContractCatalog.ResolveGroup`), consistent with how the codebase already distinguishes every other family.

**7. Neither new family is added to `CollectLayerBearingContractIds` (the `rule_input` coverage whitelist), and neither changes `IsGroupResolvable`.**
Same reasoning as #51: assembly names are never `document.Layers` keys, so the `rule_input` coverage cross-reference mechanism does not apply. Both families keep a normal `group`/`mode`/`id` shape suitable for CLI `--contract-id` selection and JSON `group` reporting (no addition to the two-family `IsGroupResolvable` exclusion list, which only covers `asmdef`/`layer_template`).

## Risks / Trade-offs

- [Risk] Confusing field reuse: `SourceType`/`ForbiddenNamespace`/`ForbiddenReferences` hold assembly simple names for these two families but type names for namespace families, and "outside allowed layers" vs. an assembly-specific message text must not be conflated. → Mitigation: assembly_allow_only uses a distinct violation message (`"outside allowed assemblies"`, not the namespace family's `"outside allowed layers"`); explicit callout in `docs/contracts/assembly-dependency.md`, matching the existing `assembly-independence.md` callout.
- [Risk] Assembly simple-name collisions (two different assemblies sharing a simple name from different probing paths) could cause a false match or miss. → Mitigation: identical identity semantics to `assembly_independence`/`ArchitectureAssemblyResolutionService`'s existing `analysis.target_assemblies` resolution; no new ambiguity introduced.
- [Risk] Direct-only detection could give false confidence that a forbidden dependency doesn't exist via an indirect path. → Mitigation: documented explicitly as a current limitation in the contract docs and YAML reference, consistent with #51's existing documented limitation for `assembly_independence`.
- [Risk] `assembly_allow_only`'s "declared target assemblies only" scoping could silently miss a violation if the offending assembly isn't listed in `analysis.target_assemblies`. → Mitigation: this mirrors the namespace-level allow-only contract's existing "not a project type" exclusion exactly, and `analysis.target_assemblies` is the same declared-assembly-universe concept the whole policy already depends on for every other assembly-axis check.
