## Context

`AddArchLinterNetCore()` already composes most of Core's constructor-injected services. Twelve production `static class` declarations remain (per `docs/internal/static-class-inventory.md` section (e)), two with hidden `private static readonly Lazy<T>` global state (`ArchitectureContractLoader._document`, `ArchitectureRepositoryRootLocator._root`). Four already have a "shadow wrapper" DI service that just forwards to the static class body (`ArchitecturePolicyDocumentLoader` → `ArchitectureContractLoader`, `ArchitectureRepositoryRootResolver` → `ArchitectureRepositoryRootLocator`, `ArchitectureProjectDiscoveryService` → `ArchitectureProjectDiscovery`, `ArchitectureAssemblyResolutionService` → `ArchitectureAssemblyResolver`). The other eight (baseline loader/merger/generator, diagnostic formatter, solution/project-file parsers, four IL/source/asmdef scanners) have no wrapper.

A structural constraint from the prior change's `core-composition-root` spec: only types under `ArchLinterNet.Core.Composition` may reference `Microsoft.Extensions.DependencyInjection` container types, and no type under `Execution`/`Scanning`/`Resolution`/`Discovery`/`Contracts` may depend on `IServiceProvider`. `ArchitectureAnalysisSession` (and its `Checking`/`Coverage`/`PolicyConsistency` partials) is a per-validation-run domain object with a public constructor used directly by tests/consumers (documented as "kept for public API stability" via `ArchitectureContractRunner"), not itself resolved through DI — it calls the four scanners and the two discovery parsers internally.

## Goals / Non-Goals

**Goals:**
- Convert all 14 listed static classes to instance classes behind interfaces; remove the two `Lazy<T>` singleton fields entirely.
- Where a shadow wrapper exists, invert it: move the real logic into the wrapper, delete the static class, rather than keep two layers.
- Register every converted service `AddSingleton` in `AddArchLinterNetCore()`.
- Preserve identical observable behavior (YAML semantics, CLI exit codes, Testing/Unity adapter output) — this is a structural refactor, not a logic change.
- Update `docs/internal/static-class-inventory.md` to reflect the final state.

**Non-Goals:**
- Threading `IServiceProvider` or DI resolution into `ArchitectureAnalysisSession`'s constructor, or changing its public signature — that's out of scope for this issue and would violate the `core-composition-root` boundary rule if done via container APIs.
- Changing scanner/parser/loader algorithms, output shapes, or exception types.
- Converting the (a)/(b)/(c)/(d) inventory classes (pure helpers, extension containers, compatibility facades) — those stay static as already classified.

## Decisions

- **Shadow-wrapped classes (`ArchitectureContractLoader`, `ArchitectureRepositoryRootLocator`, `ArchitectureProjectDiscovery`, `ArchitectureAssemblyResolver`): invert, don't layer.** The wrapper (`ArchitecturePolicyDocumentLoader`, `ArchitectureRepositoryRootResolver`, `ArchitectureProjectDiscoveryService`, `ArchitectureAssemblyResolutionService`) already has the right interface, DI registration, and constructor-injected `IArchitectureFileSystem`/`IArchitectureEnvironment`/`IArchitectureAssemblyLoader` seams. Moving the static class's method bodies into the wrapper and deleting the static class avoids keeping a redundant indirection layer, matches the issue's explicit instruction ("invert them so the wrapper owns the real logic"), and removes the `Lazy<T>` fields by construction — the wrapper is already a DI singleton, so per-instance "resolve once" caching is unnecessary (the container only builds one instance for the app's lifetime) and is dropped rather than reimplemented as an instance field, since none of the callers actually depend on cross-call caching (each call already passes its own `repositoryRoot`/`contractPath`).
- **Classes with no wrapper (baseline loader/merger/generator, diagnostic formatter, solution parser, project file parser, 4 scanners): new interface + instance class, named `IX`/`X` per the existing convention (e.g. `IArchitectureBaselineLoader`/`ArchitectureBaselineLoader`).** Same mechanical pattern as `ArchitectureContractExecutor` → `IArchitectureContractExecutor`: interface declares the same method signature the static method had (minus the class-name qualifier), implementation is `internal sealed class` (public only where an existing public constructor parameter forces it, per the precedent's CS0051 reasoning).
- **`ArchitectureBaselineLoader` and `ArchitectureBaselineMerger` fold into `ArchitectureBaselineLoadingService`.** `ArchitectureBaselineLoadingService` already calls both statics back-to-back (`LoadFromPath` then `Merge`/`MergeAndValidate`) and is the only production caller of either. Converting each into its own tiny interface would add two more DI registrations with no independent consumer; instead `ArchitectureBaselineLoader.LoadFromPath` and `ArchitectureBaselineMerger.Merge`/`MergeAndValidate` become internal instance methods directly on `ArchitectureBaselineLoadingService`, preserving the two static classes' logic and the nested `ContractGroupMerger` helper as private implementation detail. Tests that called `ArchitectureBaselineLoader`/`ArchitectureBaselineMerger` statics directly switch to constructing `ArchitectureBaselineLoadingService` with a fake `IArchitectureFileSystem`, matching how `ArchitectureBaselineLoadingServiceTests` (if any) already test the wrapper.
- **`ArchitectureBaselineGenerator` and `ArchitectureDiagnosticFormatter` get standalone interfaces (`IArchitectureBaselineGenerator`, `IArchitectureDiagnosticFormatter`)**, since each has independent production callers (`ArchitectureBaselineApplicationService` for the generator; `Program.cs`/`ArchitectureValidationResult.cs` for the formatter) that are not already-converted DI services calling into a single funnel class.
- **`ArchitectureSolutionParser`/`ArchitectureProjectFileParser` become `internal` instance classes (`IArchitectureSolutionParser`/`IArchitectureProjectFileParser`) constructor-injected into `ArchitectureProjectDiscoveryService`.** Both are already `internal` with exactly one caller (`ArchitectureProjectDiscovery`, which is folding into `ArchitectureProjectDiscoveryService` per the inversion decision above). Revised after initial review: both are registered `AddSingleton` in `AddArchLinterNetCore()` (an internal-type registration is valid since `ServiceCollectionExtensions` lives in the same assembly), with `IArchitectureProjectDiscoveryService` itself registered via a factory that resolves the two parsers from the container — this keeps them genuinely composition-root managed rather than manually `new`'d inside the service's constructor, which the spec's "registered in the Core composition root" requirement calls for even for single-caller collaborators.
- **The four scanners (`ArchitectureAsmdefScanner`, `ArchitectureSourceScanner`, `ArchitectureExternalDependencyIlScanner`, `ArchitectureIlMethodBodyScanner`) get interfaces + `AddSingleton` registration for composition-root consumers, but `ArchitectureAnalysisSession`'s internal call sites construct their own instance directly (`new ArchitectureAsmdefScanner()`, etc.) rather than receiving them via constructor.** Each scanner class has no per-instance state beyond immutable lookup tables (opcode maps, a `const` root)/regex — the "static" state was never mutable global state, just precomputed constants, so a `new()` per call is behaviorally and performance-wise identical to the prior static-field-backed lookup. This keeps `ArchitectureAnalysisSession`'s public constructor unchanged (a documented non-goal to preserve) while still eliminating every `static class` declaration and satisfying the "not static" acceptance criterion. `AsmdefValidator.cs` in `ArchLinterNet.Unity`, which currently calls `ArchitectureAsmdefScanner.FindAsmdefViolations` statically, switches to constructing its own scanner instance the same way.
- **`ArchitectureContractLoader.NormalizeToContractId` moves to a `public static` method on `ArchitecturePolicyDocumentLoader`** (the class that absorbs `ArchitectureContractLoader`'s logic per the inversion decision above, since `ArchitectureContractLoader` itself is deleted rather than kept as a non-static shell). It's a pure, stateless string-normalization utility with callers outside DI reach (other pure-helper static classes in the (a) inventory bucket, e.g. `ConditionSetResolver`); C# allows a non-static class to expose individual `static` members, so this one pure method rides along on `ArchitecturePolicyDocumentLoader` without needing its own home.
- **Register order in `ServiceCollectionExtensions`:** new registrations are added following existing dependency order (repository root resolver before policy document loader; solution/project-file parsers are private to the discovery service so they don't need top-level registration; baseline generator and diagnostic formatter added near the existing baseline/reporting registrations).

## Risks / Trade-offs

- [Missing a call site during the ~120-site test/production migration] → Mitigated by `grep`-verified enumeration of every call site per class before editing (already done in survey), and by `TreatWarningsAsErrors`/build failing to compile on any missed static reference.
- [Folding `ArchitectureBaselineLoader`/`ArchitectureBaselineMerger` into `ArchitectureBaselineLoadingService` could be seen as under-delivering on "each static class becomes its own DI service"] → The issue's acceptance criterion is "no class remains a behavior-owning static class," not "one interface per class"; documented explicitly in the inventory update and PR description as an intentional consolidation, consistent with the issue's own guidance to avoid "hand-written factory boilerplate" for classes with a single caller.
- [Constructing scanner instances inline in `ArchitectureAnalysisSession` instead of injecting them could be read as incomplete DI adoption] → Documented in this design doc and the inventory as intentional: `ArchitectureAnalysisSession` is a per-run domain object outside the composition boundary by the existing `core-composition-root` spec's own rule (no container-API dependencies outside `Composition`), and the scanners are stateless, so direct instantiation is behavior- and cost-neutral.
- [Removing the `Lazy<T>` caching in `ArchitectureContractLoader`/`ArchitectureRepositoryRootLocator` could reintroduce redundant repeated I/O for the parameterless `Load()`/`Resolve()` entry points] → Verified those parameterless entry points have no in-repo production callers (confirmed in the prior change's inventory note: "the parameterless `Load()` method is unused anywhere in-repo"); the DI singleton lifetime on the owning wrapper already gives "resolved once per app" semantics for actual callers.

## Open Questions

None — conversion pattern, registration granularity, and the `ArchitectureAnalysisSession` boundary were resolved above.
