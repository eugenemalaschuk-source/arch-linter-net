## Context

Every existing dependency/allow-only contract (`ArchitectureDependencyContract`, `ArchitectureAllowOnlyContract` in `src/ArchLinterNet.Core/Contracts/Families/*.cs`) expresses `source`/`forbidden`/`allowed` as **strings naming a declared `layers.<name>`**. Layer membership (namespace and/or `selector: { role, metadata }`) is resolved once via `ArchitectureLayerResolver`/`ArchitectureLayerTypeMatcher`, then dependency checking (`ArchitectureAnalysisSession.CheckContract`/`CheckAllowOnlyContract` in `Execution/ArchitectureAnalysisSession.Checking.cs`) walks references between the two resolved layer's type sets via `ArchitectureNamespaceViolationFinder`/`ArchitectureReferenceScanner`.

Issue #112 wants boundaries expressed **without** declaring a `layers.<name>` for every business context: a contract should be able to say "any type with role `DomainLayer` and metadata `domain: X` must not depend on any type with role `DomainLayer` and metadata `domain: Y != X`" directly. This is a different match target (a `(role, metadata)` selector value compared against another type's own resolved `(role, metadata)`, not a fixed named layer) and needs a small comparison-operator vocabulary (`exact`, `any`, `in`, `not-equal-to-source`) that today's layer selector (`ArchitectureLayerTypeMatcher.ValuesEqual`, exact/AND-only) does not have.

Every existing family follows a fixed six-touch-point pattern (model POCO → `ArchitectureContractFamilyRegistry` descriptor → checker method → violation → payload/diagnostic pair → schema/docs). This design keeps the new families inside that same pattern rather than inventing a parallel execution path.

## Goals / Non-Goals

**Goals:**
- Two new contract families, `context_dependencies` and `context_allow_only`, each with `strict_*`/`audit_*` YAML keys, registered as ordinary entries in `ArchitectureContractFamilyRegistry`.
- A `ContextSelector` model (`role` + `metadata`) used for `source`/`forbidden`/`exclude` (dependency family) and `source`/`allowed`/`exclude` (allow-only family), with metadata values supporting `exact` (literal scalar), `any` (`"*"`), `in` (YAML list of literals), and `not-equal-to-source` (`"!{source.metadata.<key>}"`).
- Reuse of `ArchitectureRoleIndex`/`ArchitectureTypeClassificationResult` for role/metadata lookup — no new classification source.
- A new diagnostic payload/diagnostic pair carrying source role/metadata, target role/metadata, and the selector kind that matched, so contextual violations are visibly distinct from namespace/layer `DependencyDiagnostic`s in both JSON and human output.
- Reuse of the existing inline `ignored_violations` mechanism unchanged.
- Registering the new families' direct role/metadata references as coverage-participating consumption, satisfying `semantic-classification-model`'s existing forward-reference, without implementing the future `scope: semantic_role` coverage variant itself (#114).

**Non-Goals:**
- Replacing or altering existing `strict_dependencies`/`strict_allow_only` (layer-name-based) contracts.
- Runtime behavior validation or semantic data-flow analysis.
- Regex-based or otherwise open-ended matching — the operator set is closed to the four named above.
- Implementing the future coverage `scope: semantic_role` variant (#114) — only exposing what it will need.
- Extending `layers.<name>.selector` itself with the new operators; that selector's exact/AND-only contract (`semantic-classification-model` spec, "Selector metadata matching is exact and AND-combined") is untouched.

## Decisions

**1. New selector type, not a reuse/extension of `ArchitectureLayerSelector`.**
`ArchitectureLayerSelector` (`Contracts/ArchitectureContractModels.cs`) is pinned by spec to exact/AND-only matching for `layers.<name>.selector`. Extending it with `any`/`in`/`not-equal-to-source` would change behavior for existing layer declarations. Introduce a distinct `ArchitectureContextSelector` (`Role` + `Metadata` where each metadata value is a raw YAML scalar/sequence, deferred-parsed into an operator) used only by the two new contract families. A new `ArchitectureContextSelectorMatcher` performs matching; it may delegate role-equality/decimal-coercion helpers by extracting the shared pieces of `ArchitectureLayerTypeMatcher` into a small internal helper rather than duplicating them outright.

**2. Metadata value grammar is a closed, string-prefix-based mini-grammar, consistent with the existing `metadata.<key>` extraction grammar's "checked in fixed order" style** (`semantic-classification-model` spec, "Metadata extraction syntax is fixed and deterministic"):
- A YAML sequence value → `in` (match if the type's resolved metadata value equals any list entry, using the same cross-domain equality as today's `ValuesEqual`).
- The literal string `"*"` → `any` (matches any resolved value, including a missing key resolving to "matched" for existence purposes — matches issue's `domain: "*"` source-selector example).
- A string matching `^!\{source\.metadata\.[A-Za-z0-9_]+\}$` → `not-equal-to-source`, resolved at match time against the *source type's own* resolved metadata value for that key (only valid on `forbidden`/`allowed`/`exclude` selectors, never on `source` itself, since "source" has no other source to reference).
- Any other scalar → `exact` literal, compared via the existing string/bool/decimal coercion rules.
This keeps parsing deterministic and avoids a general expression language, per the issue's "no unrestricted regex-based matching" non-goal.

**3. Missing source or target role/metadata is a non-match, not an error.** If a candidate type has no resolved role (`ArchitectureRoleIndex.TryGetRole` returns false) it cannot satisfy any contextual selector — consistent with `ArchitectureLayerTypeMatcher`'s existing behavior for selector-backed layers. If a `not-equal-to-source` reference names a metadata key the source type does not have, the operator does not match (there is no source value to be unequal to) — this is exercised directly by the issue's required "missing source metadata"/"missing target metadata" test cases.

**4. Checker logic mirrors `CheckContract`/`CheckAllowOnlyContract` but scans by selector match instead of resolved layer membership.** Add `CheckContextDependencyContract`/`CheckContextAllowOnlyContract` to `ArchitectureAnalysisSession.Checking.cs` (or a new partial, e.g. `ArchitectureAnalysisSession.CheckingContext.cs`, if the existing file is already large) that: (a) enumerate classified types matching the `source` selector, (b) for each, resolve its outgoing references via the existing `ArchitectureReferenceScanner`, (c) for each referenced type, resolve its role/metadata and test against `forbidden`/`allowed` selectors and `exclude`, honoring `not-equal-to-source` against the specific source type currently being checked (evaluation is therefore per-source-instance, not global). `exclude` selectors are applied identically to how existing contracts treat exclusions — a match against any `exclude` selector suppresses the candidate target entirely before violation/allow evaluation.

**5. Two contract families, registered independently.** `context_dependencies` (forbidden-list semantics, mirroring `dependency`) and `context_allow_only` (allowed-list semantics, mirroring `allow_only`) are registered as two separate `ArchitectureContractFamilyDescriptor` entries appended to `ArchitectureContractFamilyRegistry.All`, each with `IncludeInContractEnumeration = true` and no `ConfigurationContributor` layer-name collection (contextual contracts reference no `layers.<name>`, so there is nothing to add to the layer-name collector — this is itself the mechanism that must be reconciled with future coverage work per Decision 7).

**6. New diagnostic pair: `ContextDependencyPayload`/`ContextDependencyDiagnostic`** (dependency family) and **`ContextAllowOnlyPayload`/`ContextAllowOnlyDiagnostic`** (allow-only family), following the existing `DependencyPayload`/`DependencyDiagnostic` pattern (`Model/DependencyPayload.cs`, `Model/DependencyDiagnostic.cs`) and a new `ArchitectureDiagnosticKind` value per family. Each diagnostic carries: `SourceRole`, `SourceMetadata` (the resolved subset relevant to the matching selector), `TargetRole`, `TargetMetadata`, and `MatchedSelector` (`"forbidden"`/`"allowed"`/`"exclude"`) so JSON/human output can distinguish *which* selector produced the finding — satisfying the issue's "selector evidence" and "clearly distinguish contextual contract violations from existing namespace/layer dependency violations" acceptance criteria. `ArchitectureDiagnosticMapper.FromViolation` requires no change beyond the existing `violation.Payload?.ToDiagnostic(violation)` dispatch already being payload-polymorphic.

**7. Coverage-consumption registration is a marker, not new coverage logic.** `semantic-classification-model`'s existing requirement ("Consumption is not hard-coded to layers.<name>.selector alone") only obligates this change to make the consumed `(role, metadata)` pairs discoverable identically to a selector match — not to implement `scope: semantic_role` itself (that is #114's job, explicitly out of scope here per the issue's non-goals). Concretely: record each contextual contract's `source`/`forbidden`/`allowed`/`exclude` selectors' `(role, metadata-key)` pairs on the classification session (e.g. a session-level `RegisteredContextualConsumers` collection populated during contract loading/registration, parallel to how `ConfigurationContributor` already records layer-name consumption for the existing families) so a later coverage change can query it without needing to know the internal shape of contextual contracts. This change adds the recording hook and the delta-spec requirement; it does not add a consuming coverage contract.

**8. Baseline support: added if it fits cleanly, else explicitly deferred.** `ArchitectureBaselineModels.cs`'s per-group switch-case (`GetGroup`/`SetGroup`/`GroupNames`) is a flat enumeration keyed by contract-family group name; adding `context_dependencies`/`context_allow_only` cases mechanically mirrors the `dependency`/`allow_only` cases already there, using the same `ArchitectureIgnoredViolation`-shaped baseline entries (baseline entries key on `SourceType`/`ForbiddenReference`, which remain meaningful for contextual contracts since violations still resolve to concrete types). Given this mirrors an existing mechanical pattern with no shape mismatch found during design, baseline support is included in this change's scope rather than deferred.

**9. Schema additions are structurally parallel to `$defs.dependencyContract`/`$defs.allowOnlyContract`.** Add `$defs.contextSelector` (sibling to, not reusing, `$defs.selector`, since it needs the operator vocabulary: `role` required string, `metadata` a map whose values may be a scalar, the literal `"*"`, a `not-equal-to-source` string pattern, or an array), `$defs.contextDependencyContract`, `$defs.contextAllowOnlyContract`, and four new `contracts.*` properties (`strict_context_dependencies`, `audit_context_dependencies`, `strict_context_allow_only`, `audit_context_allow_only`) under `$defs.contracts`.

## Risks / Trade-offs

- **Selector proliferation** (`ArchitectureLayerSelector` vs. `ArchitectureContextSelector`) risks confusing authors about which selector shape applies where → Mitigated by keeping the two types structurally distinct (different YAML contexts: `layers.<name>.selector` vs. `contracts.*context_*.source/forbidden/allowed/exclude`) and documenting both clearly with a comparison table in `docs/policy-format/semantic-classification.md`.
- **Per-source-instance evaluation of `not-equal-to-source`** is more expensive than the existing layer-membership-set precomputation (existing contracts precompute layer membership once; contextual dependency checking must re-evaluate `forbidden`/`allowed` selectors against each source type's own metadata) → Mitigated by caching each *non-cross-referencing* selector's matching type set once per contract (only `not-equal-to-source` operators require per-source-type re-evaluation, and even then only for the specific metadata key referenced).
- **Coverage-consumption marker (Decision 7) is speculative** until #114 actually consumes it → Mitigated by keeping the marker a plain data collection with no coupling to a specific future coverage contract shape; if #114 needs a different shape, only the recording call site (not the contextual contract feature itself) needs revisiting.
- **`exclude` semantics ambiguity**: existing families have no `exclude` field at all (they use `ignored_violations` post-hoc instead) → This design treats `exclude` as a pre-match filter on candidate *targets* (per the issue's YAML example, `exclude: [{ role: SharedKernel }]`), distinct from `ignored_violations` which remains a post-violation suppression keyed on concrete type names. Documented explicitly in the new capability specs to avoid confusion between the two mechanisms.
