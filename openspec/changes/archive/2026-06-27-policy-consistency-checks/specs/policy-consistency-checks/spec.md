## ADDED Requirements

### Requirement: Policy consistency pass runs separately from code scanning
The validation pipeline SHALL run a policy-consistency check as a distinct step after `configuration_check` and before `contract_checks`, operating only on the loaded and layer-template-expanded policy document (`ArchitectureContractGroups`), without scanning any code.

#### Scenario: Consistency check runs before contracts execute
- **WHEN** `ArchitectureValidationService.Validate` runs against a policy
- **THEN** the policy-consistency check SHALL execute after configuration validation and before any `ArchitectureContractExecutor.Execute` contract runs

#### Scenario: Consistency check does not require code
- **WHEN** the policy-consistency check runs
- **THEN** it SHALL NOT require resolved code violations from contract execution to produce its findings, except for layer-overlap detection which reuses the same type/namespace resolution already performed for the empty-layer configuration check

### Requirement: Duplicate contract IDs are detected
The system SHALL detect duplicate contract IDs across strict and audit contract families, including IDs generated by layer-template expansion, and SHALL report each duplicate as a policy-consistency finding before any contract-ID-based selection occurs.

#### Scenario: Duplicate explicit IDs
- **WHEN** two contracts in the same or different families declare the same `Id`
- **THEN** the policy-consistency check SHALL report a finding identifying both contract names and the shared ID

#### Scenario: Duplicate expanded template IDs
- **WHEN** a layer template expands into two or more contracts that compute the same ID (e.g. via `{templateIdBase}/{containerIdBase}` collision)
- **THEN** the policy-consistency check SHALL report a finding identifying the template and the colliding expanded IDs

### Requirement: Allow-only versus forbidden dependency conflicts are detected
The system SHALL detect when an allow-only contract permits a dependency from a source layer to a target layer while another contract forbids that same source/target layer pair, and SHALL report the conflict with both contract IDs/names.

#### Scenario: Direct allow/forbid conflict
- **WHEN** an allow-only contract allows `domain -> application` and a dependency contract forbids `domain -> application`
- **THEN** the policy-consistency check SHALL report a finding identifying both contracts, the source layer, and the target layer

### Requirement: Independence versus explicit dependency conflicts are detected
The system SHALL detect when an independence contract requires two layers to be mutually independent while another contract explicitly allows or orders a dependency between the same two layers, and SHALL report the conflict.

#### Scenario: Independence contradicted by an explicit allowed dependency
- **WHEN** an independence contract declares layers `a` and `b` mutually independent and a separate contract explicitly allows `a -> b`
- **THEN** the policy-consistency check SHALL report a finding identifying the independence contract, the conflicting dependency contract, and both layers

### Requirement: Protected-importer conflicts are detected
The system SHALL detect when a protected surface's `allowed_importers` includes an importer that another strict forbidden/protected rule forbids for the same surface, and SHALL report the conflict.

#### Scenario: Allowed importer explicitly forbidden elsewhere
- **WHEN** a protected contract's `allowed_importers` includes layer `x` for a protected surface, and a strict forbidden/protected rule forbids `x` from importing that same surface
- **THEN** the policy-consistency check SHALL report a finding identifying both contracts, the protected surface, and the importer layer

### Requirement: Overlapping layer definitions are detected
The system SHALL detect when a concrete type or namespace is matched by more than one internal layer definition without an explicit documented allowance, and SHALL report the conflict with the matched layer names and a representative concrete type/namespace.

#### Scenario: Two internal layers match the same type
- **WHEN** a concrete type's namespace matches the patterns of two internal (non-external) layers and no documented allowance reconciles the overlap
- **THEN** the policy-consistency check SHALL report a finding listing both layer names and that type as the representative concrete type/namespace

#### Scenario: External layer overlap is not a conflict
- **WHEN** a concrete type's namespace matches both an internal layer and a layer marked `external: true`
- **THEN** the policy-consistency check SHALL NOT report a layer-overlap finding for that type

### Requirement: Unreachable contracts are detected
The system SHALL detect contracts whose declared source or target layer can never match any type because the layer's own pattern definition is structurally empty or mutually exclusive, and SHALL report the contract as unreachable.

#### Scenario: Contract references a structurally impossible layer
- **WHEN** a contract's source or target layer has include/exclude namespace patterns that cancel out so no namespace can ever match
- **THEN** the policy-consistency check SHALL report a finding identifying the contract and the unreachable layer

### Requirement: Policy consistency severity is configurable
The system SHALL support `analysis.policy_consistency` with values `error`, `warn`, or `off`, defaulting to `error` when unset. When `error`, any policy-consistency finding SHALL cause `ValidationOutcome.Passed` to be `false`. When `warn`, findings SHALL be reported but SHALL NOT affect `Passed`. When `off`, the policy-consistency check SHALL still run but SHALL NOT produce any diagnostics or affect `Passed`.

#### Scenario: Default severity fails validation
- **WHEN** a policy with a duplicate contract ID is validated and `analysis.policy_consistency` is not set
- **THEN** `ValidationOutcome.Passed` SHALL be `false`

#### Scenario: Warn severity reports without failing
- **WHEN** a policy with an allow/forbid conflict is validated and `analysis.policy_consistency: warn`
- **THEN** the finding SHALL be present in the diagnostics output and `ValidationOutcome.Passed` SHALL NOT be affected by it

#### Scenario: Off severity suppresses findings
- **WHEN** a policy with a layer overlap is validated and `analysis.policy_consistency: off`
- **THEN** no policy-consistency diagnostic SHALL be produced and `ValidationOutcome.Passed` SHALL NOT be affected by it

#### Scenario: Invalid severity value rejected
- **WHEN** `analysis.policy_consistency` is set to a value other than `error`, `warn`, or `off`
- **THEN** the system SHALL throw `InvalidOperationException`

### Requirement: Findings are deterministic
The system SHALL emit policy-consistency findings in a deterministic order across repeated runs against the same policy document, independent of dictionary/hash-set enumeration order.

#### Scenario: Repeated validation produces identical findings
- **WHEN** the same policy document is validated twice
- **THEN** the ordered list of policy-consistency findings SHALL be identical both times

### Requirement: Existing valid policies are unaffected
A policy containing no internal contradictions SHALL produce zero policy-consistency findings and SHALL pass validation exactly as it did before this change.

#### Scenario: Green policy passes unchanged
- **WHEN** a policy with no duplicate IDs, no allow/forbid conflicts, no independence conflicts, no protected-importer conflicts, no layer overlaps, and no unreachable contracts is validated
- **THEN** `ValidationOutcome.Passed` SHALL be unaffected by the policy-consistency check
